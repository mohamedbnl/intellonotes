# Project Overview

## Detected project name

- Package name: `intellonotes` (from `package.json`).
- User-facing branding: “IntelloNotes” (e.g. `src/app/[locale]/layout.tsx`, `src/components/layout/Header.tsx`).

## What the app appears to do

This repository implements a **role-based programming courses platform** with:

- A **public course catalog** of approved courses, with search and filters.
- **Email/password authentication**.
- A structured **5-axis learning experience** per course (axes 1–5), combining:
  - optional PDF material (uploaded by professors)
  - lesson text stored in the database
  - a quiz per axis (axis 5 enforces a passing score)
  - an in-browser code playground for Python/JavaScript

Three roles exist in the codebase (DB schema + route guards):

- **student**: dashboard + learning
- **professor**: create/edit courses + earnings/withdrawals
- **admin**: approve/reject/suspend courses, confirm/reject purchases, process/reject withdrawals

## Main objective / product purpose

Enable professors to create course content and submit it for admin review; enable students to buy/unlock courses (via manual payment confirmation) and then complete the course through axis-based progression with progress tracking.

## Utilities / “services”

There is no separate “services” layer; the closest equivalents are:

- DB query helpers in `src/lib/db/queries.ts`.

- Utility modules like `src/lib/utils/commission.ts` and `src/lib/utils/quiz-grader.ts`.

## Data flow (concrete, code-based)

Read path (common pattern):

1. A server `page.tsx` reads locale and calls `setRequestLocale(locale)`.
2. It fetches translations via `getTranslations(...)`.
3. It fetches DB data via synchronous Drizzle+SQLite query helpers in `src/lib/db/queries.ts`.
4. It renders server output and passes props to client components (interactive UI).

Write path (mutations):

- Client components call server actions from `src/lib/actions/*` (e.g. create course, save axis content, create purchase, save quiz result).
- Some features use API routes:
  - `/api/upload-pdf` for file upload/delete
  - `/api/pdf/*` for streaming files
  - `/api/purchase-status` for polling purchase status
- Mutations generally trigger `revalidatePath(...)` server-side and `router.refresh()` client-side.

## Frontend/backend boundaries (server/client split)

Server-only:

- `src/lib/db/*` (SQLite connection, schema, queries)
- `src/lib/actions/*` (`"use server"`)
- `src/app/api/*/route.ts` handlers
- Server components (most `src/app/[locale]/**/page.tsx`)

Client-only:

- Interactive components marked `"use client"` (forms, editors, learning UI)
- Client hooks (e.g. `src/hooks/useAuth.ts`, `src/hooks/usePyodide.ts`)
- Dynamic imports with `ssr: false` for heavy browser-only modules (Monaco, react-pdf viewer)

# Database and Data Layer

## Whether Drizzle is used and how

Drizzle is used for the runtime DB:

- Connection: `src/lib/db/index.ts` uses `better-sqlite3` and configures WAL + foreign keys.
- Schema: `src/lib/db/schema.ts` defines tables and enums for users/courses/lessons/quizzes/purchases/progress/withdrawals.
- Queries: `src/lib/db/queries.ts` exposes read and write helpers used by pages and server actions.

Environment variable usage:

- `DATABASE_PATH` is read in `src/lib/db/index.ts` and in `drizzle.config.ts`.

## Whether Supabase is used and how

Supabase is **not used in runtime code** under `src/`:

- No Supabase client imports are present in `src/`.
- `.env.example` contains no Supabase keys.

However, the repository includes Supabase assets:

- `supabase/migrations/*.sql`: Postgres schema + enums + RLS policies (`00008_rls_policies.sql`).
- `supabase/seed.sql`: seed data for Postgres.
- `supabase/config.toml`: Supabase CLI local development configuration.

## Migrations / schema / seed setup

SQLite (implemented):

- Migration output: `drizzle/`
  - `drizzle/0000_glamorous_sage.sql` creates all tables and indexes.
  - `drizzle/0001_whole_phil_sheldon.sql` adds `courses.rejection_reason`.
- Seed script: `src/lib/db/seed.ts`.
- Package scripts:
  - `db:migrate`: `drizzle-kit migrate`
  - `db:seed`: `tsx src/lib/db/seed.ts`
  - `db:reset`: deletes `intellonotes.db`, migrates, then seeds.

Supabase/Postgres (present, not wired to runtime):

- Migration files exist and define RLS policies, but there is no runtime linkage.

## Probable database flow (runtime)

1. App starts and opens SQLite at `DATABASE_PATH` (or `intellonotes.db`).
2. Server pages/actions read/write synchronously through Drizzle/better-sqlite3.
3. UI refreshes via `revalidatePath(...)` and `router.refresh()`.

## Ambiguity between Drizzle and Supabase if present

The repo contains two competing “stories”:

- Implemented: SQLite + Drizzle + NextAuth credentials + local file storage.
- Documented/artifacts: Supabase Postgres/RLS schema and Supabase-oriented instructions in `REQUIREMENTS.md`.

There is no code that bridges the implemented runtime to Supabase.

- **Supabase vs runtime mismatch**: `supabase/` and `REQUIREMENTS.md` describe Supabase/Postgres/RLS, but runtime uses SQLite/Drizzle + NextAuth credentials; `.env.example` contains no Supabase env vars.

# End-to-End Functional Summary

1. Middleware routes users into `/<locale>/...` (`fr` or `ar`), and `src/app/[locale]/layout.tsx` provides the HTML shell, translations, and header navigation.
2. Users browse the catalog (`/[locale]`) and open a course detail page (`/[locale]/courses/[courseId]`).
3. Users can register/login using credentials; session carries `user.id` and `user.role`.
4. A student can create a pending purchase request for a course; an admin confirms it from the admin payments queue.
5. After confirmation, the student enters the learning route (`/[locale]/courses/[courseId]/learn`), which verifies purchase, loads lessons/quizzes, and tracks progress across 5 axes until course completion.
6. Professors create and edit draft courses (PDF + axis lesson/quiz content) and submit them for admin review; admins approve/reject/suspend, and approved courses appear in the public catalog.
